## MVP Plan: `scriptorium discord`

### Core concept

A separate process — `scriptorium discord` — that acts as a Discord frontend to the architect. It follows the same pattern as `scriptorium ask` and `scriptorium plan`: a standalone command that reads and writes plan branch state. The orchestrator doesn't know or care that Discord exists.

### How it works

1. `scriptorium discord` starts a guildy gateway connection in a blocking loop.
2. A whitelisted user sends a message in the configured channel.
3. The bot invokes the architect with the message as input (same as `scriptorium ask`, but the prompt comes from Discord instead of stdin).
4. The architect reads the spec, iteration log, and project state — same context it always has.
5. The architect's response gets posted back to the channel.
6. If the architect modifies the spec or creates tickets, it does so on the plan branch as normal.

The orchestrator picks up spec/ticket changes on its next tick, same as if someone had run `scriptorium plan` manually. No coupling between the two processes.

### Config

A `discord` section in `scriptorium.json`:

```json
{
  "discord": {
    "enabled": false,
    "channelId": "123456789",
    "allowedUsers": ["215660018010161152"]
  }
}
```

- **channelId**: Scopes the bot to a single channel. Messages in other channels are ignored.
- **allowedUsers**: List of Discord user IDs. Messages from anyone else are ignored.
- **Token**: Comes from the `DISCORD_TOKEN` environment variable, never stored in config.

### Slash commands

Four commands that require no LLM calls — they just read or write orchestrator state directly:

- `/status` — Report current iteration, tickets in flight, merge queue contents, whether the orchestrator is running. Reads from `.scriptorium/` and the plan branch.
- `/queue` — Show the current ticket queue with statuses. Formatted read of plan branch state.
- `/pause` — Write a pause flag to `.scriptorium/`. The orchestrator stops picking up new work; in-flight agents finish but nothing new starts.
- `/resume` — Remove the pause flag.

### Chat messages

Non-slash messages from allowed users go to the architect. Each message is treated as a standalone architect invocation (stateless — no Discord chat history carried between messages). The architect already has rich context from the plan branch, spec, and iteration log. A message like "focus on pathfinding" doesn't need chat history to be actionable.

The architect responds conversationally and can optionally make spec or ticket changes as part of its response.

### Dependencies

- guildy (already used in other monofuel projects)
- No new dependencies for the orchestrator — the bot is a separate process.

### Out of scope (future ideas)

- **Threaded conversations**: Create a Discord thread per architect interaction for multi-turn context. Would let users have back-and-forth planning discussions.
- **Interactive orchestrator control**: Beyond pause/resume — abort tickets, reprioritize areas, skip review for specific PRs.
- **Reaction-based approvals**: Thumbs-up on a bot message to approve a risky merge or accept a score regression.
- **Notification channel**: A read-only channel where the bot posts iteration summaries, merge results, and agent failures. Separate from the interactive command channel.
- **Multi-channel setup**: Per-area channels, separate notification vs command channels.
- **Mattermost/Slack adapters**: Same interface, pluggable transport backend.
- **Live streaming**: Bot posts real-time agent output to a thread as a coding agent works.
- **Bot-to-bot**: Multiple scriptorium instances in the same server, different projects, occasionally commenting on each other's work.
- **Webhook mode**: Discord webhooks for notifications-only (lighter weight, no persistent gateway connection).
