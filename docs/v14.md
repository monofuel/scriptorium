# v14 plan

Focus: code quality passes and spec hygiene.

## 1. Spec consolidation

### Problem

Scriptorium's own `spec.md` has accumulated versioned sections â€” `V2 Features`,
`V3 Features`, `V4 Features`, `V5 Features`, `V13 Features` â€” each with their
own "Known Limitations" and "Acceptance Criteria" blocks. The spec reads like a
changelog or release notes, not a blueprint.

On fresh projects this doesn't happen because the architect writes a clean spec
from scratch. But on long-running projects (like scriptorium itself), each
orchestrator run appends a new version heading rather than integrating the new
capabilities into the existing structure.

### Desired behavior

The spec should be a **current-state blueprint** â€” a single coherent document
describing what the system does *now*. Version history belongs in git commits,
tags, and `docs/vN.md` files, not in the spec itself.

After consolidation, the spec should:

- Have one flat list of numbered requirements organized by topic (CLI, planning,
  orchestrator, agents, merge queue, config, etc.) â€” not by version.
- Remove all "Known Limitations" and "Acceptance Criteria" sections entirely.
  These are work-log artifacts. The spec defines what the project *does*, not
  what's broken or how to verify a release.
- Drop version headings entirely.

### Approach

Two parts:

**Part A: Update the architect prompt to prevent recurrence.**
The architect's system prompt (or the spec template used during `scriptorium
plan`) should explicitly instruct:
- Integrate new capabilities into existing spec sections by topic, never append
  versioned blocks.
- The spec is a current-state blueprint, not a changelog. Do not add "Known
  Limitations", "Acceptance Criteria", "V*N* Features", or similar
  release-oriented sections.
- Version history belongs in git tags and `docs/vN.md` files.

Build and test the prompt changes before proceeding to Part B.

**Part B: One-time spec cleanup via `scriptorium plan`.**
With the improved prompt in place, prompt the architect interactively to
consolidate the spec. The architect already has full access to `spec.md` â€”
just tell it what to fix:
- Merge all versioned feature sections into a flat topic-based structure.
- Remove all "Known Limitations" and "Acceptance Criteria" sections â€” these are
  work-log artifacts, not part of a project specification. The spec describes
  what the system *is*, not what's left to do or how to verify a release.
- Drop version headings entirely.

## 2. Review agent cleanup pass

### Problem

Coding agents iterate toward passing tests but leave behind cruft â€” dead code,
abandoned first-attempt approaches, inconsistent naming, unnecessary imports.
This is especially common after context compaction, where the agent loses track
of earlier iterations. The result is "slop on top of slop" that accumulates over
time.

### Desired behavior

Before a PR is merged, the review agent should flag (and optionally fix) quality
issues beyond just "do the tests pass":

- Unreachable code or unused imports introduced by the PR.
- Leftover artifacts from abandoned approaches (commented-out code, TODO
  comments that reference completed work, variables assigned but never read).
- Changes unrelated to the ticket's stated goal â€” but without being too
  aggressive, since incidental fixes (e.g. a bug found in an edge case) are
  sometimes legitimate.

### Open questions

- Should the review agent fix issues itself, or just flag them for a separate
  cleanup pass?
- How do we prevent the review agent from being overly aggressive and rejecting
  legitimate incidental improvements?
- Should there be a dedicated "cleanup" agent role, or is this just an expansion
  of the review agent's responsibilities?

## 3. Per-role log persistence

### Problem

Agent logs are split inconsistently and some are lost entirely:

- **Review agent** logs land in the ticket worktree
  (`<worktree>/.scriptorium/logs/<ticketId>/`). When the worktree is cleaned up
  after merge, the logs are deleted. The only surviving record is the
  interleaved orchestrator log (`run_*.log`), which only captures tool-call
  summaries, not the full JSONL stream.
- **Coding agent execution** logs have the same problem â€” they're written to the
  worktree, not the repo root.
- **Architect, manager, and prediction** logs already persist correctly under
  `<repo>/.scriptorium/logs/`, but use inconsistent directory names
  (`plan-spec/`, `architect-areas/`, `manager/`, `<ticketId>-prediction/`).

### Desired log structure

```
.scriptorium/logs/
  orchestrator/              # run_*.log (existing, unchanged)
  architect/
    spec/                    # architect spec runs
    areas/                   # architect area generation
  manager/
    <areaId>/                # per-area manager runs
  coder/
    <ticketId>/              # coding agent execution
      attempt-01.jsonl
      attempt-02.jsonl
  prediction/
    <ticketId>/              # difficulty predictions
  review/
    <ticketId>/              # review agent runs
      attempt-01.jsonl
```

### Changes

1. **Review agent**: Set `logRoot` in the `AgentRunRequest` to
   `repoPath/.scriptorium/logs/review` so JSONL logs persist after worktree
   cleanup. Currently `merge_queue.nim` does not set `logRoot`, so the harness
   defaults to `<workingDir>/.scriptorium/logs/` (the worktree).

2. **Coding agent execution**: After the agent finishes (or on worktree
   cleanup), copy the JSONL logs from the worktree to
   `repoPath/.scriptorium/logs/coder/<ticketId>/`. Alternatively, set `logRoot`
   to the repo root at request time â€” but this requires care since the agent's
   `workingDir` is the worktree.

3. **Consistent directory naming**: Rename existing log directories to match
   role names. Migrate: `plan-spec/` â†’ `architect/spec/`,
   `architect-areas/` â†’ `architect/areas/`,
   `<ticketId>-prediction/` â†’ `prediction/<ticketId>/`. The `manager/` directory
   is already fine.

## 4. Drift detection audit

### Problem

Over time the codebase drifts from both `spec.md` and `AGENTS.md`. Nobody
notices until a human reads the code and wonders why it doesn't match. By then
the drift has compounded across many tickets.

### Desired behavior

A read-only audit agent runs periodically (e.g. after every merge, or every N
merges) in the background. It does not block the merge queue or modify any
code â€” it only produces a report.

The audit checks two things:

1. **Spec compliance**: Read `spec.md` and sample the codebase. Report
   divergences â€” features the spec describes that the code doesn't implement,
   code behaviors not reflected in the spec, contradictions between spec
   requirements and actual behavior.

2. **AGENTS.md compliance**: Read `AGENTS.md` and the recent merge diff (or
   a broader code sample). Report violations of project conventions â€” naming,
   logging, error handling, file organization, whatever AGENTS.md specifies.

Reports are written to `.scriptorium/logs/audit/` as timestamped markdown
files. They're informational â€” the human or a future ticket decides what to
act on.

### Trigger: merge queue drain

The audit runs when the merge queue fully drains (goes idle after processing
one or more items). This is event-driven and avoids all the bad alternatives:

- **Not per-merge**: Auditing after each of 7 merges in a burst is wasteful.
  Waiting until the queue drains means one audit covers the whole burst.
- **Not timer-based**: If scriptorium sits idle for hours, a timer wastes
  credits auditing unchanged code.
- **Not manual**: If it requires human action, it never gets run.

The audit batches naturally â€” it covers all merges since the last audit report.
The orchestrator tracks a "last audited commit" hash. When the merge queue
drains, it compares the current default branch HEAD against the last audited
commit. If they differ, it spawns the audit agent. If nothing merged (queue
drained because items were reopened, not merged), no audit runs.

A secondary trigger: **spec change**. If the architect rewrites `spec.md`, the
existing code may now contradict the new spec. An audit after spec changes
catches "spec says X but code still does old-Y" immediately.

### Design considerations

- Runs as a background agent using a shared pool slot (like managers/coders),
  so it doesn't need a dedicated concurrency mechanism.
- Should be cheap: use a smaller/faster model (Haiku), limit scope to the
  cumulative diff since the last audit plus `spec.md` and `AGENTS.md`, keep
  the prompt focused.
- Output format: markdown report written to `.scriptorium/logs/audit/` with
  a timestamp. Sections for spec drift and AGENTS.md violations, each item
  citing the relevant rule and the offending code location.
- The audit agent should also be invocable on demand via `scriptorium audit`
  for humans who want an immediate check.

### Configuration

The audit agent needs its own entry in `scriptorium.json` under `agents`,
like the other agent roles:

```json
{
  "agents": {
    "architect": { "model": "claude-opus-4-6" },
    "coding":    { "model": "claude-opus-4-6" },
    "manager":   { "model": "claude-opus-4-6" },
    "reviewer":  { "model": "claude-opus-4-6" },
    "audit":     { "model": "claude-haiku-4-5-20251001" }
  }
}
```

This lets users configure the audit agent's model, harness, and reasoning
effort independently. The default should be a cheap/fast model (Haiku) since
the audit is informational and doesn't need to write code.

The `AgentConfigs` type in `config.nim` gains an `audit` field with a
corresponding `DefaultAuditModel` constant. The `resolveModel()` function
already handles Haiku model IDs.

## 5. Compaction-resilient agent context

### Problem

Claude Code compacts conversation history when it approaches context limits.
After compaction, the agent loses track of AGENTS.md rules, project
conventions, and earlier decisions. This causes the agent to drift from
conventions mid-ticket â€” introducing patterns that were explicitly forbidden,
reverting to defaults the project has overridden, etc.

### Approach

Several complementary fixes:

1. **CLAUDE.md reinforcement**: The most critical AGENTS.md rules should be
   present in CLAUDE.md (or `.claude/` config files). Claude Code reloads
   CLAUDE.md on every invocation and after compaction, so these rules survive
   context loss. Scriptorium already syncs AGENTS.md â€” consider also syncing
   a subset of rules to CLAUDE.md, or ensuring AGENTS.md is loaded in a
   compaction-resistant way.

2. **Continuation prompt re-injection**: When scriptorium builds a continuation
   prompt for a retry or timeout recovery, re-inject the AGENTS.md rules (or a
   condensed version) into the continuation text. The continuation prompt
   builder (`buildContinuationPrompt` in `harness_claude_code.nim`) already
   has access to the working directory â€” it could read AGENTS.md and append a
   "Reminder: follow these project rules" section.

3. **Smaller, focused AGENTS.md**: A large AGENTS.md is more likely to be
   compacted away or ignored. Consider splitting into per-area rule files that
   are only included when relevant, reducing the total context load.

## 6. Review agent enforces spec and AGENTS.md

### Problem

Currently the review agent is a rubber stamp â€” in the v13 run, every single
ticket was approved with zero rejections. The review agent checks nothing
beyond whether the coding agent called `submit_pr`. Even when the coding agent
drifts from AGENTS.md conventions or introduces code that contradicts the spec,
the review agent approves it.

### Desired behavior

The review agent should enforce both `spec.md` and `AGENTS.md` as part of its
review. Specifically:

- **AGENTS.md enforcement**: The review prompt should include AGENTS.md content.
  The reviewer checks the diff against project conventions and flags violations.
  This is the last line of defense â€” even if the coding agent went off the rails
  after compaction, the review agent catches it.

- **Spec enforcement**: The review prompt should include relevant spec sections
  (at minimum the section related to the ticket's area). The reviewer checks
  that the implementation matches the spec and flags contradictions.

- **Graduated response**: Not every violation warrants rejection. Minor style
  issues should be logged as warnings in the review notes but still approve.
  Substantive violations (wrong behavior, spec contradictions, convention
  violations that affect correctness) should request changes.

### Changes

- Update the review agent prompt (`review_agent.md` or equivalent) to include
  AGENTS.md and relevant spec sections as context.
- Add graduated severity to review decisions: approve-with-warnings vs.
  request-changes.
- Log review reasoning (not just the decision) so violations are traceable
  in the review logs.
