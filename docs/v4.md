# v4 plan

- merge reviewing

- I use the terms "pr" and "merge request" but this is actually a fully local tool and we are not in fact opening PRs on github. submit_pr is just a literal request to scriptorium to merge the changes into the master branch.


## TODO list

- [ ] review agent
  - once an agent calls submit_pr, another agent should review the PR.
  - submit_pr should run tests before doing anything- if tests fail, it should just fail the function and tell the agent to fix the tests without actually submitting the PR.
  - once reviewed, it should either approve the PR or request changes.
  - to keep things simple, we could close the previous coding agent (who submitted the pr) and then start a new review agent codex session. if changes are requested, we should just start a new codex coding session with the same ticket and the feedback to finish the work.
    - maybe in the future we could intentionally stall the coding agent during review, but that sounds complicated?
- [ ] commit health cache
  - the master health check (`make test` + `make integration-test`) runs on every orchestrator startup and after every merge, even if that exact commit was already tested.
  - add a cache that records which commits have been tested and their pass/fail result.
  - store on the plan branch (e.g. `health/<commit-hash>.json` or a single `health/cache.json`) so it survives container restarts.
  - on startup or after merge, look up the current master HEAD in the cache — if it was already tested and healthy, skip the check entirely.
  - cache entries should include: commit hash, timestamp, test exit codes, test wall time.
  - invalidation: cache entries are keyed by commit hash so they're naturally immutable. pruning old entries is optional.
