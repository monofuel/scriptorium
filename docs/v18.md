# scriptorium dashboard

- web dashboard for observing and monitoring scriptorium orchestrator state
- especially important with v16 loops running unattended and v17 discord adding another surface

## MVP Plan

### Architecture

`scriptorium dashboard` — a mummy web server that reads orchestrator state from `.scriptorium/` and the plan branch. Separate process, no coupling to the orchestrator. Same pattern as `scriptorium discord`.

- **Server**: Mummy HTTP server with WebSocket support
- **Communication**: WebSocket-first, following the andrewlytics pattern. Single WS connection at `/ws`. Server pushes HTML fragments using htmx `hx-swap-oob` for multi-region updates.
- **Rendering**: Server-side HTML generation in Nim (procs that return HTML strings). Nimponents for client-side interactive components where needed (charts, live log viewer). Minimal JS — just compiled nimponents and htmx.
- **Port**: 7777

### Config

```json
{
  "dashboard": {
    "enabled": false,
    "port": 7777
  }
}
```

### Polling model

The dashboard server polls `.scriptorium/` and the plan branch on a short interval (1-2s) and pushes diffs to connected WebSocket clients. No sub-second latency needed — knowing what happened 1-2 seconds ago is fine for monitoring.

### Prerequisite: structured event log

The orchestrator currently spreads state across scattered files. The dashboard needs a unified event stream. The orchestrator should write events to `.scriptorium/events.jsonl` (append-only JSON lines):

- Agent started (ticket, area, slot)
- Agent finished (ticket, success/failure, duration, token usage)
- Merge completed (ticket, success/conflict)
- Iteration boundary (iteration number, feedback output)
- Orchestrator started/stopped/paused

The dashboard reads this file for the recent events feed and historical data. The plan branch still provides current ticket/area state; the event log provides the timeline.

### Pages

#### Overview (main page)

- Current iteration number (v16 loop mode)
- Orchestrator status: running / paused / stopped
- Active agents: slots used out of maxAgents
- Recent events feed (last ~20 events)
- Quick stats: tickets completed today, tickets in queue, merge queue depth

#### Iteration timeline

- Table showing each iteration: number, strategy summary, feedback result, score delta
- Click an iteration to expand: full iteration log entry, tickets created, tickets completed
- Future: nimponents canvas chart for score over time

#### Ticket board

- Active tickets grouped by area
- Status indicators: queued, in-progress, review, merged, failed
- Click a ticket to expand: branch name, agent session log tail, PR status
- Color coding by status

#### Agent monitor

- Live view of running agents: which ticket, how long running, slot usage
- Recent completions: success/failure, duration, token usage
- Failed sessions with expandable error logs

#### Merge queue

- Items waiting for review/merge
- Current item being processed
- Recent merge history: success/conflict/failure

### Data sources

Everything already exists on disk — the dashboard reads it:

- `.scriptorium/` — lock files, agent PIDs, orchestrator state, event log
- Plan branch — spec, areas, tickets, iteration log
- Git log — recent commits, merge history
- Agent session logs
- `scriptorium.json` — config (maxAgents, loop settings)

### Technology

- **mummy**: HTTP server + WebSocket (already a dependency)
- **htmx**: Client-side declarative updates via WebSocket
- **nimponents**: Web components for interactive elements (charts, log viewer)
- **Server-side HTML**: Nim procs generating HTML strings, pushed over WebSocket with OOB swaps

Most of the dashboard is server-rendered HTML. Nimponents is only needed for the score chart and live log viewer — everything else is htmx swaps.

### MVP scope

1. `scriptorium dashboard` command — mummy server with WebSocket
2. Structured event log (`.scriptorium/events.jsonl`) added to the orchestrator
3. Overview page: orchestrator status, active agents, recent events
4. Ticket board with statuses
5. Iteration timeline as a table

### Out of scope (future ideas)

- Score charts over time (nimponents canvas component)
- Live agent log streaming
- Cost tracking (tokens per iteration, cost estimates, budget limits)
- Authentication (local tool, trust the network for now)
- Mobile-responsive layout
- Embedded terminal for running scriptorium commands from the dashboard
- Multi-project view (multiple scriptorium instances on one dashboard)
