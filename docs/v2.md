## Known Issues

### Stale worktree metadata in Docker (BLOCKER)

**Status:** Blocks every tick. Not transient — repeats indefinitely until manually cleaned.

**Symptom:** The manager generates tickets successfully, but the write guard check immediately after fails with:

```
[ERROR] tick N failed: git diff --name-only --relative failed while checking write guards:
fatal: not a git repository: /workspace/.git/worktrees/plan
```

This repeats every tick. No tickets are ever committed to the plan branch. The orchestrator burns API credits regenerating tickets that are thrown away every time.

**Root cause chain:**

1. The orchestrator calls `withLockedPlanWorktree`, which creates a git worktree at `/tmp/scriptorium/<hash>/worktrees/plan` inside the container.
2. Git writes tracking metadata to `/workspace/.git/worktrees/plan/gitdir`, which points back at the `/tmp` path (e.g. `/tmp/scriptorium/workspace-304b40cf6073a3f1/worktrees/plan/.git`).
3. The manager agent (claude-code) runs in the worktree and writes ticket files. This works fine.
4. The write guard runs `git diff --name-only --relative` inside the worktree to verify the manager only wrote to allowed paths.
5. But the worktree's `.git` file references `/workspace/.git/worktrees/plan`, and that entry's `gitdir` points to a `/tmp` path. The `/tmp` path still exists at this point (same container), but something about the git metadata chain is broken — `git diff` fails with `fatal: not a git repository`.
6. The tick fails. The worktree gets cleaned up (the `/tmp` dir is removed), but `.git/worktrees/plan/` on the host mount **persists**.
7. On the next tick, `addWorktreeWithRecovery` sees the stale `.git/worktrees/plan` entry. It should prune it, but apparently doesn't — or the prune doesn't work because the metadata points to a path that no longer exists.
8. The cycle repeats.

**Why self-healing doesn't work:** The earlier diagnosis assumed `addWorktreeWithRecovery` would handle stale worktrees. In practice, it doesn't prune `.git/worktrees/plan` between ticks. The stale metadata accumulates and blocks all subsequent attempts.

**Manual workaround:** `rm -rf .git/worktrees/plan && git worktree prune` on the host. This unblocks the next tick, but the problem recurs immediately because the same sequence happens again.

**Fix needed in `addWorktreeWithRecovery` (orchestrator.nim):** Before creating a new plan worktree, run `git worktree prune` in the repo root to clean up entries pointing to nonexistent paths. This is a one-line fix but needs to happen before the `git worktree add` call.

**Impact:** The orchestrator cannot make any progress past the architect phase. Manager tickets are generated but never committed. Coding agents never start. This is the highest priority fix for self-hosted Docker operation.

### Docker container runs as non-root (resolved)

claude-code refuses `--dangerously-skip-permissions` when running as root. Fixed by adding a `scriptorium` user (uid 1000) in the Dockerfile and switching to `USER scriptorium`.

### nim.cfg host paths don't resolve in container (resolved)

The host repo's `nim.cfg` contains nimby package paths like `/home/monofuel/.nimby/pkgs/...` which don't exist inside the container. Fixed by copying `.nimble` and `.nimby` from root to the scriptorium user in the Dockerfile, and adding an `entrypoint.sh` that runs `nimby sync -g nimby.lock` in the workspace on startup.

### Git identity missing for non-root user (resolved)

The new `scriptorium` container user had no git `user.email`/`user.name` configured, causing plan branch commits to fail. Fixed by adding `git config --global` in the Dockerfile.

### Manager spawns one session per area (resolved)

`runManagerTickets` looped over areas and spawned a separate claude-code session for each one. Each session redundantly explored the entire codebase from scratch, taking ~30 minutes total. Fixed by batching all areas into a single manager session using a new `manager_tickets_batch.md` prompt template with an `{{AREAS_BLOCK}}` placeholder. One runner call now handles all areas sequentially.

### Manager creates tickets for already-implemented code (resolved)

On first run against an existing codebase, the manager generated open tickets for functionality that was already complete. Fixed at the prompt level: the batch template now instructs the agent to check `{{PROJECT_REPO_PATH}}` for existing implementations before creating tickets, and to place fully-implemented areas into `tickets/done/` instead of `tickets/open/`.

### Done tickets not tracked by area suppression (resolved)

`collectActiveTicketAreas` only scanned `tickets/open/` and `tickets/in-progress/`, so areas with done tickets would get re-evaluated on every future run. Fixed by adding `PlanTicketsDoneDir` to the scan loop. The write guard and git add were also expanded to allow `tickets/done/` writes from the manager.

### All agent tiers using Opus (resolved)

Architect, manager, and coding agents all defaulted to `claude-opus-4-6`, wasting cost on tasks Sonnet handles well. Fixed by updating `scriptorium.json` to use `claude-sonnet-4-6` for manager and coding agents, keeping Opus only for the architect.
