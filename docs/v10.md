# v10 plan

Focus: integrate Typoi as an alternative coding agent backend alongside Codex and Claude Code.

## Background

Scriptorium currently supports two agent harnesses: `codex` and `claude-code`. The
`harnessTypoi` enum value already exists in `config.nim` but the dispatcher in
`agent_runner.nim` raises "not implemented". Typoi (the CLI agent harness from the
Typos project) already supports streaming responses, tool calling, and multiple AI
providers (OpenAI, Anthropic, Bedrock, LM Studio), but lacks two things scriptorium
requires: MCP client support (for `submit_pr` / `submit_review`) and a dotfile
configuration layer (for storing auth and defaults outside of CLI flags).

Work is split into two phases: first prepare Typoi itself, then wire it into
scriptorium.

## Phase 1 — Typoi improvements (in Typos repo)

### 1.1 Home dotfile configuration (`~/.typos/config.json`)

Typoi currently takes all configuration via CLI flags and environment variables.
Add a layered config system with profile support:

- On startup, load `~/.typos/config.json` if it exists.
- Merge with any project-local `.typos.json` if present.
- CLI flags override file config; file config overrides defaults.
- Support named profiles selected via `--profile <name>` (e.g. `--profile aws`,
  `--profile openai`). A `default` profile is used when no flag is given.
- Config schema:
  ```json
  {
    "defaultProfile": "openai",
    "profiles": {
      "openai": {
        "provider": "openai",
        "model": "gpt-5.1-codex-mini",
        "apiKeyEnv": "OPENAI_API_KEY",
        "baseUrl": "",
        "reasoningEffort": "medium"
      },
      "aws": {
        "provider": "bedrock",
        "model": "openai.gpt-oss-20b",
        "apiKeyEnv": "BEDROCK_API_KEY",
        "baseUrl": "",
        "reasoningEffort": "medium"
      },
      "anthropic": {
        "provider": "anthropic",
        "model": "claude-sonnet-4-6",
        "apiKeyEnv": "ANTHROPIC_API_KEY",
        "baseUrl": "",
        "reasoningEffort": "medium"
      }
    }
  }
  ```
- `apiKeyEnv` names the environment variable holding the key — the config file
  never contains secrets directly. This follows the same pattern as Codex and
  Claude Code.

### 1.2 MCP client support (using MCPort)

Typoi needs to act as an MCP client so scriptorium can expose `submit_pr` and
`submit_review` tools to it. Use the MCPort library (already used by scriptorium
for its MCP server) for the client implementation. MCPort's MCP client is lightly
tested, so fixes to MCPort may be needed as part of this work.

- Accept `--mcp-server-url <url>` CLI flag to connect to an HTTP MCP server.
- On startup (when flag is set), use MCPort's client to call `tools/list` and
  merge discovered tools into the existing tool registry.
- When the model invokes an MCP tool, proxy the call via MCPort's `tools/call`
  and return the result to the model.
- MCP tools should appear alongside native tools — the model doesn't need to
  know which tools are local vs remote.
- Only HTTP transport is needed (scriptorium runs an HTTP MCP server).
- Any MCPort client bugs discovered should be fixed upstream in MCPort.

### 1.3 JSONL stream output for scriptorium

Typoi already has `--json-stream` output. Verify that the JSONL events emitted
are compatible with what scriptorium's harness parser expects:

- Heartbeat events at a configurable interval.
- Tool-call events (name + arguments) so scriptorium can track progress.
- Final message extraction (already supported via `--output-last-message`).
- Exit code semantics: 0 = success, non-zero = failure.

Adjust event format if needed to match the `AgentStreamEvent` kinds scriptorium
uses: `heartbeat`, `reasoning`, `tool`, `status`, `message`.

## Phase 2 — Scriptorium integration

### 2.1 Implement `harness_typoi.nim`

Create `src/scriptorium/harness_typoi.nim` following the pattern of
`harness_codex.nim` and `harness_claude_code.nim`:

- Define `TypoiRunRequest` / `TypoiRunResult` types.
- Build the `typoi` command line from the request:
  - `--prompt` with the ticket prompt
  - `--provider` / `--model` / `--reasoning-effort` from config
  - `--mcp-server-url` pointing to scriptorium's MCP endpoint
  - `--json-stream` for structured output
  - `--yolo` for read+write tool access
  - `--output-last-message` for final message capture
  - Working directory set to the ticket worktree
- Parse JSONL stdout and emit `AgentStreamEvent` via the `onEvent` callback.
- Track timeouts (hard, no-output, progress) using the same logic as other
  harnesses.
- Extract final message from the output file or last message event.

### 2.2 Wire up the dispatcher

In `agent_runner.nim`, replace the `harnessTypoi` raise with a real dispatch:

```nim
of harnessTypoi:
  let typoiResult = runTypoi(TypoiRunRequest(...))
  result = AgentRunResult(backend: harnessTypoi, ...)
```

### 2.3 Configuration and defaults

- `inferHarness()` in `config.nim` already maps unknown model prefixes to
  `harnessTypoi` — verify this is still the desired fallback.
- Add a `typoi` section to `scriptorium.json` schema if needed (e.g. custom
  binary path, extra CLI flags).
- Document supported models and provider setup in the README or a doc file.

### 2.4 Tests

- Unit test: `harness_typoi.nim` command-line construction and event parsing.
- Integration test: round-trip with a mock Typoi binary that emits canned JSONL
  and calls `submit_pr` via the MCP endpoint.
- Verify existing orchestrator tests work with `harnessTypoi` by swapping the
  harness in config.
