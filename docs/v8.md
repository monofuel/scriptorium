# v8 plan

Focus: crash recovery, resilience, and deployment reproducibility.

## Startup recovery and consistency checks

The orchestrator currently has no startup recovery. If it crashes mid-operation,
the next run starts fresh with no awareness of stale state. Specific gaps:

**Orphaned worktrees:** `cleanupStaleTicketWorktrees` exists but is never
called at startup. After a crash, `/tmp` worktrees from the previous run
persist. Git worktree locks may be held by dead processes, blocking new ticket
assignment. New worktree creation for the same ticket will fail.

**Stale agent processes:** If the orchestrator crashes while coding agents are
running, those `codex` or `claude` CLI processes continue running. They hold
file locks on worktrees and may still be writing to branches. On restart, the
orchestrator has no knowledge of these orphaned processes and may try to assign
the same tickets or create conflicting worktrees.

**Inconsistent plan branch state:** Multi-step operations (file move + git add
+ git commit) have crash windows. For example, if the orchestrator crashes
after `moveFile(ticketPath, planPath / doneRelPath)` but before the git commit,
the ticket file is in `done/` on disk but not committed. The plan branch's
working tree and git index disagree.

**Stale merge queue:** If a crash happens during `processMergeQueue` after the
ff-only or no-ff merge to master succeeds but before the plan branch records
the completion, master has the code but the plan branch still shows the ticket
as in-progress with a pending merge queue entry.

### Proposed startup sequence

Add a `recoverFromCrash` proc called at the beginning of `runOrchestrator`:

1. **Clean stale worktrees:** Call `cleanupStaleTicketWorktrees(repoPath)` to
   prune worktrees that don't correspond to in-progress tickets.

2. **Kill orphaned agent processes:** Scan `/tmp/scriptorium-*/` worktrees for
   lock files, check if the holding PID is still alive. If the PID is dead,
   remove the lock. If alive but not a known scriptorium process, log a warning.

3. **Reconcile plan branch:** Open a plan worktree and check for uncommitted
   changes (dirty working tree). If found, either commit them (if they look
   like a completed transition) or reset them (if they look like a partial
   operation). Log what was recovered.

4. **Validate merge queue vs master:** For each ticket in the merge queue, check
   if the ticket's branch is already merged into master. If so, complete the
   plan branch transition (move to done/, commit). This handles the crash
   window where master was updated but plan wasn't.

5. **Log recovery summary:** Report what was cleaned up so the operator knows
   what state was recovered.

## Atomic plan branch transitions

The current pattern for ticket state transitions is:

```
writeFile(ticketPath, updatedContent)     # update file content
moveFile(ticketPath, newPath)             # move to new directory
removeFile(queuePath)                     # clean up queue entry
writeFile(activePath, "")                 # clear active marker
gitRun(planPath, "add", "-A", ...)        # stage everything
gitRun(planPath, "commit", "-m", ...)     # commit
```

If any step after the first `writeFile` fails or the process crashes, the plan
branch is in an inconsistent state. Git's staging area is the only transaction
boundary, and it's not used atomically.

### Proposed approach

Wrap multi-step plan transitions in a helper that:
1. Records the intended operation (source path, dest path, content) in a
   journal file on the plan branch before starting
2. Executes the filesystem operations
3. Commits everything (including the journal entry)
4. Removes the journal file in a follow-up commit

On startup recovery, if a journal file exists, the recovery proc can either
replay or roll back the incomplete transition based on what state the filesystem
is actually in.

This is similar to a write-ahead log. The journal file makes the intent
explicit so recovery doesn't have to guess.

## Docker image reproducibility

The Dockerfile has no version pinning:

```dockerfile
FROM gitea.solution-nine.monofuel.dev/monolab/monolab-nim:latest

RUN npm install --global @openai/codex @anthropic-ai/claude-code
```

Both the base image (`:latest`) and npm packages (no `@version`) will resolve
to whatever is current at build time. Builds on different days produce different
images. A breaking change in `@openai/codex` or `@anthropic-ai/claude-code`
could silently break the orchestrator.

### Proposed changes

1. Pin the base image to a specific tag or digest:
   ```dockerfile
   FROM gitea.solution-nine.monofuel.dev/monolab/monolab-nim:2.2.4@sha256:...
   ```

2. Pin npm packages to specific versions:
   ```dockerfile
   RUN npm install --global @openai/codex@0.1.2 @anthropic-ai/claude-code@1.0.20
   ```

3. Add a CI job or Makefile target that checks for available updates and opens
   a ticket or PR when new versions are released. This keeps the pins fresh
   without sacrificing reproducibility.

4. Consider printing installed versions in the entrypoint script for
   traceability in logs.

## Agent process crash recovery

Beyond orchestrator crashes, individual agent processes can crash or hang:

**Hung agents:** The no-output timeout (default 5 min) and hard timeout
(default 4 hours) catch most hangs, but a process that produces occasional
garbage output can evade the no-output timeout while making no real progress.
Consider adding a "progress check" — if the agent hasn't called any MCP tools
or produced meaningful output (e.g. tool calls, file edits) within a window,
treat it as stalled.

**Zombie worktrees after agent crash:** If an agent process dies mid-edit, its
worktree may have uncommitted changes, partial file writes, or corrupt state.
The orchestrator should check worktree health before reusing it for a retry:
- Run `git status` to detect dirty state
- Run `git diff` to assess whether partial work is salvageable
- If corrupt, delete and recreate the worktree from the branch

**Branch state after agent crash:** If an agent crashes after committing to its
branch but before calling `submit_pr`, the work is saved in git but the
orchestrator doesn't know about it. On retry, the next agent attempt starts
from the branch tip (which includes the prior work), which is correct. But
the merge queue is never notified. Consider checking for committed-but-not-
submitted work at the start of each attempt.
