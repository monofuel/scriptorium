# v13 plan

Focus: parallel managers and improved default concurrency.

## Background

Scriptorium currently runs the manager as a single blocking call per tick —
`runManagerTickets()` builds one batch prompt containing all areas and sends it
to a single agent invocation using `manager_tickets_batch.md`. This is a
bottleneck when there are many areas, and it doesn't match the natural
parallelism of the system: areas are independent, so their managers should be
too.

There are actually two manager code paths today:

- `runManagerTickets()` — the production path. Collects all areas into one
  batch prompt, holds the plan worktree lock for the entire duration, and lets
  one agent write tickets for all areas in a single pass.
- `syncTicketsFromAreas()` — an older per-area loop that takes a
  `ManagerTicketGenerator` callback and processes areas sequentially. Also
  holds the lock the entire time.

Both paths are sequential and hold the lock too broadly. v13 replaces them
with a concurrent per-area model.

Coding agents already support parallelism via `maxAgents`, but the default is
1 (serial mode). Most users should benefit from parallel execution out of the
box.

## Concurrency model

The orchestrator has four agent roles with different concurrency rules:

### Strictly sequential (no parallelism)

- **Architect** — reads `spec.md`, generates/updates `areas/*.md`. Runs at
  most once per tick, only when the spec hash changes. Protected by the plan
  worktree lock. Must complete before managers run.

- **Review / Merge** — evaluates completed work, runs tests, and merges to
  the default branch. Processes one merge queue item at a time, sequentially.
  This is intentional: merging is the point where we guarantee the default
  branch stays healthy. Every merge candidate is tested against the current
  default branch head before fast-forwarding. Sequential merging means we
  never merge broken code, even if two agents finish simultaneously.

### Parallel (shared slot pool)

- **Manager** — generates tickets for a single area. Each manager invocation
  handles one area using `manager_tickets.md`. Multiple managers can run in
  parallel when there are multiple areas that need tickets.

- **Coding agent** — implements a single ticket in an isolated worktree.
  Multiple coding agents can run in parallel when there are tickets in
  independent areas.

Managers and coding agents share the same `maxAgents` slot pool. If
`maxAgents` is 4 and 2 managers are running, only 2 slots remain for coding
agents (and vice versa). This keeps total resource usage bounded without
needing separate limits for each role.

### Interleaved execution

Managers and coders are interleaved across ticks — the orchestrator does not
wait for all managers to finish before starting coders. If area A's manager
finishes and produces tickets while area B's manager is still running, those
tickets can be assigned to coding agents immediately on the next tick.

### Merge conflicts

Parallel coding agents working in different areas may still produce merge
conflicts — shared utility files, imports, config changes, etc. This is
expected and acceptable. The sequential merge process catches conflicts:

1. Merge default branch into ticket branch before testing.
2. If there's a conflict, the merge fails.
3. The ticket is sent back for another coding attempt with the conflict as
   context.
4. The coding agent resolves the conflict in its next attempt.

Area-based separation makes conflicts less likely, but the system is designed
to handle them gracefully when they occur.

## Changes

### 1. Default `maxAgents` to 4

Change the default from 1 to 4 in `config.nim`. This gives new projects
parallel execution out of the box.

Users can still set `maxAgents: 1` in `scriptorium.json` for serial mode.

Files: `src/scriptorium/config.nim`

### 2. Generalize the slot pool

The current slot/thread management in `coding_agent.nim` is specific to
coding agents. Extract it into a shared `agent_pool.nim` module that both
managers and coders use.

**Current state:** `AgentSlot` is defined in `shared_state.nim` with
coding-agent-specific fields (ticketId, branch, worktree). The slot list,
channel, thread management, and functions (`startAgentAsync`,
`checkCompletedAgents`, `runningAgentCount`, `emptySlotCount`,
`joinAllAgentThreads`) all live in `coding_agent.nim`.

**New design:**

- Add a `role` field to `AgentSlot`: `AgentRole = enum arCoder, arManager`.
- Manager slots use `areaId` instead of `ticketId` (or a shared `id` field).
  Manager slots have no branch or worktree — they produce results in memory.
- `AgentCompletionResult` gains a `role` field and an optional
  `managerResult` (the generated ticket documents) alongside the existing
  coding result fields.
- `startAgentAsync()` accepts a role and a generic worker proc. Coding agents
  pass their existing `executeAssignedTicket` wrapper. Manager agents pass a
  new `executeManagerForArea` wrapper.
- `checkCompletedAgents()` returns completions tagged with role so the
  orchestrator can dispatch accordingly.

Files: `src/scriptorium/agent_pool.nim` (new), `src/scriptorium/coding_agent.nim`,
`src/scriptorium/shared_state.nim`, `src/scriptorium/manager_agent.nim`

### 3. Split manager into per-area invocations

Replace `runManagerTickets()` (batch path) with per-area concurrent
invocations.

**Current flow (blocking, batch):**
1. Acquire plan worktree lock.
2. Collect all areas needing tickets.
3. Build one batch prompt with all areas.
4. Run one agent — it writes ticket files directly to the plan worktree.
5. Commit all tickets.
6. Release lock.

**New flow (concurrent, per-area):**
1. Orchestrator briefly acquires plan lock to read areas needing tickets and
   their content, then releases the lock.
2. For each area, if a slot is available, spawn a manager agent thread. The
   agent uses `manager_tickets.md` (single-area prompt) and generates ticket
   content **in memory** — it does not write to the plan worktree.
3. Manager thread sends results (list of `TicketDocument`) back through the
   shared agent result channel.
4. On the next tick, `checkCompletedAgents()` picks up manager completions.
5. The orchestrator (main thread) acquires the plan lock, calls
   `writeTicketsForArea()` to persist tickets, commits, and releases the lock.

This means the agent thread never touches the plan worktree — only the main
thread writes, eliminating concurrent write conflicts entirely.

**Template change:** The production path switches from
`manager_tickets_batch.md` to `manager_tickets.md`. The batch template is
retired. `syncTicketsFromAreas()` is also removed — the new thread-based
per-area flow replaces it.

Files: `src/scriptorium/manager_agent.nim`, `src/scriptorium/orchestrator.nim`,
`src/scriptorium/prompt_builders.nim`

### 4. Restructure the orchestrator tick

The current tick is strictly sequential: architect → manager (blocking) →
assign tickets → coding. With parallel managers, the tick becomes:

1. Poll completed agents (managers + coders) via `checkCompletedAgents()`.
   - For completed managers: acquire plan lock → write tickets → commit →
     release. Log results.
   - For completed coders: handle as today (move ticket, queue merge, etc).
2. Check backoff / health.
3. Run architect (sequential, if spec changed).
4. Read areas needing tickets (brief plan lock).
5. For each area needing tickets, if slots available, start a manager agent.
6. For each assignable ticket, if slots available, start a coding agent.
7. Sleep.

Key difference: steps 5 and 6 compete for the same slot pool, so the
orchestrator should prioritize based on what's most useful. A reasonable
default: managers first (they unblock future coding work), then coders.

Files: `src/scriptorium/orchestrator.nim`

### 5. Manager-specific plan branch locking

With parallel managers, the plan worktree lock strategy needs refinement:

- **Reading areas:** Brief lock to snapshot area content at the start of the
  tick. Can be done once for all areas, not per-manager.
- **Agent execution:** No lock needed. Manager agents run in threads and
  produce ticket content in memory.
- **Writing tickets:** Main thread acquires lock, writes tickets for one
  completed manager, commits, releases. Each completed manager's write is a
  separate lock acquisition — short and non-blocking.

The architect still holds the lock for its full duration (it reads spec and
writes area files). This is fine because architect is sequential and runs
before managers are spawned.

Files: `src/scriptorium/manager_agent.nim`, `src/scriptorium/lock_management.nim`
