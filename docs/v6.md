### Key Findings

**1. Duration predictions are consistently ~2x too high.**
Almost every ticket finished in roughly half the predicted time. Even "accurate"
difficulty predictions overestimate duration: 0043 predicted 45min but took
16.8min; 0052 predicted 35min but took 17.2min. The prediction prompt has no
calibration data telling it how fast Claude actually works.

**2. Difficulty vs duration mismatch.**
The predictor conflates code complexity with time. Tickets marked "hard" often
just needed more attempts (stalls), not more wall time per attempt. The actual
coding time per attempt is quite consistent at 2-8 minutes.

**3. Underestimation correlates with attempt count, not code complexity.**
The "underestimated" tickets (0040, 0045, 0048, 0051) all required 2+ attempts.
The code itself wasn't harder — the agent just stalled or hit tool/merge issues.

### Prediction Prompt Issues

The prediction prompt (`src/scriptorium/prompts/ticket_prediction.md`) is
bare-bones. It asks for "estimated wall time in minutes" but gives no baseline
for how fast the agent works. A human might take 35 minutes on a medium task;
Claude typically finishes in 8-15 minutes of coding time (plus test/retry
overhead).

Recommendations:
- Add calibration context with actual historical timing data, e.g.:
  "The coding agent typically completes 2-8 minutes of coding per attempt.
  Easy tasks take ~10min wall time, medium ~15-20min, hard ~25-40min.
  Factor in that stall retries add ~5min each."
- Separate difficulty prediction from duration prediction — difficulty should
  reflect code complexity; duration should account for expected attempt count
  and test overhead.
- Feed the predictor a few examples from completed tickets with actual wall
  times to anchor its estimates.

### Agent Run Issues

**`submit_pr` MCP tool unavailability (0045, 0051):**
On first attempts, the agent sometimes reports the MCP tool isn't available.
This caused unnecessary stalls. Likely a timing/registration race condition
in how the orchestrator starts the MCP server relative to when the agent tries
to discover tools.

**Merge conflicts (0051):**
When master diverged (from manual commits outside the orchestrator), the agent's
worktree couldn't fast-forward merge. Error: "Your local changes would be
overwritten by merge." This is an orchestrator-level issue — the merge queue
should handle rebasing or retrying when master has moved.

**No missing tools noted:**
No complaints about ripgrep, grep, or similar being unavailable. The agents
have what they need tool-wise.

**No codebase confusion noted:**
The agents didn't flag confusing code areas. The main friction was operational
(stalls, tool timing, merge conflicts) rather than comprehension.

### 1. Prediction Prompt Recalibration

**File:** `src/scriptorium/prompts/ticket_prediction.md`

Prompt-only change, no Nim code touched. The original prompt was a minimal
template that asked for "estimated wall time" with no reference data. The
predictor had no way to know Claude finishes most tasks in 8-15 minutes.

Changes:
- Added a **Calibration Data** section with baseline timings derived from v5
  actuals: coding time per attempt (2-8 min), overhead per attempt (3-5 min),
  stall retry cost (~5-10 min), and wall-time ranges per difficulty level.
- Rewrote **Instructions** to explicitly separate `predicted_difficulty`
  (code complexity, files touched, integration risk) from
  `predicted_duration_minutes` (wall time = coding × attempts + overhead).
- Replaced the single generic example with three calibrated few-shot examples
  anchored to real v5 tickets:
  - Easy: config change, ~11 min (based on ticket 0041)
  - Medium: cache persistence, ~17 min (based on ticket 0043)
  - Hard: parallel assignment, ~25 min (based on ticket 0048)

Template placeholders (`{{TICKET_CONTENT}}`, `{{AREA_CONTENT}}`,
`{{SPEC_SUMMARY}}`) unchanged. No changes to `buildPredictionPrompt` or
`prompt_catalog.nim`. Existing unit test passes.

### 2. MCP Server Readiness Probe

**File:** `src/scriptorium/orchestrator.nim`

The race condition: `runOrchestratorLoop` spawned the HTTP server thread and
immediately entered the main loop. Agents dispatched in early ticks could
attempt MCP tool discovery before the server was listening.

Changes:
- Added `std/httpclient` to imports.
- Added constants `ServerReadyTimeoutMs = 5000` and
  `ServerReadyPollIntervalMs = 50`.
- Added `waitForServerReady` proc that polls `http://{address}:{port}/mcp`
  with a 500ms per-request timeout until it gets any response, or the 5s
  deadline expires. Logs success or warns and proceeds on timeout.
- Inserted `waitForServerReady` call between `createThread(serverThread, ...)`
  and `runOrchestratorMainLoop(...)` so the main loop doesn't start until
  the MCP endpoint is confirmed reachable.

The probe is intentionally lenient — on timeout it warns and continues rather
than aborting, so the orchestrator still works in degraded environments.

### 3. Merge Queue --no-ff Fallback

**File:** `src/scriptorium/orchestrator.nim` (`processMergeQueue`)

The original code attempted only `git merge --ff-only` into master. When
master had diverged (e.g. manual commits outside the orchestrator), this
failed and the ticket was reopened — even though quality checks had already
passed on the ticket branch.

The new flow in `withMasterWorktree`:
1. Try `git merge --ff-only` — if it succeeds, return immediately (fast path
   unchanged).
2. If ff-only fails, log and attempt `git merge --no-ff --no-edit`.
3. If the --no-ff merge itself fails (conflict), run `git merge --abort` and
   return the failure.
4. If the --no-ff merge succeeds, re-run `runRequiredQualityChecks` on the
   merged master state to verify nothing broke from the merge commit.
5. If quality checks fail, `git reset --hard HEAD~1` reverts the merge commit
   and returns the failure.
6. If everything passes, log success and return.

The failure step message was updated to "ff-only and no-ff both failed" to
reflect both strategies were attempted. Existing ff-only-success paths are
unaffected (they hit the early return).
