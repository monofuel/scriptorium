## coding nitpicks

- the coding agent keeps doing 'python style imports" where all the imports are on individual import lines. this is not right.
  - nim style imports use a single import block, and we personally want them sorted.

bad import example:

```
import unittest
import "../src/game"
import "../src/systems/input"
import "../src/entities/character"
```

good import example:

```
import
  std/[unittest, strformat, strutils],
  boxy, silky,
  game, input, character,
```
import order should be "std/[] imports first, then libraries, then local imports.

- imports do not need quotes. eg: you can use `../common`, you don't need `"../common"`

- next related nit, we should have a tests/config.nims with `--path:"../src"` as the contents.
  - this enables tests to automatically import src/ files without having to use an absolute ../src/ path.


- next, commenting standards.
  - all (or maybe most) functions should have a nimdoc comment.
  - nimdoc comments start INSIDE the proc block, and start with ##

- also, this is more pedantic, but I would like to ensure that ALL comments are complete sentences with a capital first letter and the comment ends with a period.
  - this makes comments good to read in nimdocs, but also, I found this makes a good sentinel to see if the agent is smart enough to follow the rules.

good nimdoc example:
```
proc sumOfMultiples(limit: int): int =
  ## Calculate the sum of all multiples of 3 or 5 below the limit.
  var total = 0
  for i in 1..<limit:
    if i mod 3 == 0 or i mod 5 == 0:
      total += i
  return total
```



- if an object property is the same name as a nim keyword, you must wrap it in backticks
```
  DeleteModelResponse* = ref object
    id*: string
    `object`*: string
    deleted*: bool
```

- variables should be grouped together. consts first, then let, then var.
- consts should be capitalized camelCase.
- let and var should be regular camelcase.
- do not place 'magic variables' in the code, instead make them a const and pull them up to the top of the file.
- for example:
```
const
  Version = "0.1.0"
  Model = "llama3.2:1b"
let
  embeddingModel = "nomic-embed-text"
var
  total = 0
  count = 0
```

- object vs ref object
  - I usually prefer ref objects, but Claude seems to often pick object. not a big deal but this could be confusing in certain cases. objects are 'pass by value' which can be a problem for big objects or if you try to assign properties. ref objects are 'pass by reference' which is more efficient, but can be more confusing if you don't expect spooky property changes at a distance.
