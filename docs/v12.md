# cleanup pass


- brainstorming
  - often times working with LLM agents with codex / claude, it is useful to do a 'cleanup pass' of a feature branch before it is fully ready to merge.
    - weird stuff creeps in from iterations. like maybe the agent goes in one direction, then changes to go another direction, but doesn't clean up the first pass (commonly happens during compaction)
    - we should probably have the review agent suss out changes that are unrelated to the PR, but also not be too aggressive in case there are some legit fixes (eg: maybe we found a bug in an edge case, we don't need it anymore but it's still good to fix)
  - Currently we just throw slop on top of slop as long as tests pass.
    - I wonder how we could introduce "improvement passes" without being too crazy. eg: large file refactors, fix bad patterns.
    - the easy answer is "do it right the first time" but sometimes systems get scrapped, or sometimes they grow in strange directions, you can't predict every case.
