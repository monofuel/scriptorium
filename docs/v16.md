### minecraft bot

- nebulous open-ended problem
- say we have a base spec for a minecraft bot as a server plugin
  - say this bot is a mix of static code (java plugin), LLM logic (llm structured responses, llm tool calling, etc) and maybe also runtime logic (eg: javascript or lua scripting)
  - eg: we may have a top level loop that says "build house" then the llm decides where to put the house, and then the llm builds a lua script to define how the house should be built.
- the architect could continuously improve the code for the bot
- so the bot would continuously be running on a minecraft server, and the architect would watch over it and continuously make changes
- could it actually beat the game and reach the end?
- could it even make a house without dieing on the first night?

### metta cogames bot

- https://github.com/metta-AI/metta
- give a scriptorium instance a fresh repo and a base spec (perhaps one of our template nim bots) and tell it to iterate on getting a higher score.

- some assumptions:
  - have a make target to upload the bot
  - have scripts to fetch bot scores
  - have make targets to test the bot locally in various situations

- I already have a decent bot with automation for uploading, fetching scores, testing. easy to iterate on

- the "loop" would be telling the architect to keep iterating on getting the score up, forever. forever!
- some ideas
  - maybe different managers could test changes independantly and gate results on if it actually gets the score up? issues that hurt the score could get aborted or closed as "wontfix" by the review agent or manager?
  - some core high level loop of "upload bot" "wait for scores". we can test locally a lot but the real scores have to come from the web leaderboards
  - I think that the "queue drain" moments are key? architect makes changes. queue is filled. queue drains. no work left. this is when the architect should handle uploading bot, waiting for scores, and strategizing for the next version.
