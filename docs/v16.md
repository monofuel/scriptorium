## MVP Plan

### Core concept

One new concept: **iterations**. When the queue drains, instead of stopping, the orchestrator runs a feedback step and hands the results to the architect for the next iteration.

### Config additions

A `loop` section in `scriptorium.json`:

```json
{
  "loop": {
    "enabled": true,
    "goal": "Maximize tokens/sec on 7900 XTX. Currently ~160, target 358+.",
    "feedback": "make bench",
    "maxIterations": 0
  }
}
```

- **goal**: Free-text description of what we're trying to achieve. The architect reads this every iteration.
- **feedback**: Shell command that runs at queue drain. Its stdout becomes context for the next iteration. Can output anything — numbers, logs, observations, structured text. Whatever is useful for the problem.
- **maxIterations**: 0 means infinite.

No metric parsing, no threshold config, no special modes per problem type. The entire loop surface is these three fields.

### The iteration log

A file on the plan branch: `iteration_log.md`. Append-only. Each entry written by the architect at the start of an iteration, after reviewing feedback from the previous iteration:

```markdown
## Iteration 3
**Feedback output**: 182 t/s, memory bandwidth 78% utilized
**Assessment**: Score up from 171. Memory coalescing changes worked.
**Strategy**: Continue memory optimization. Try shared memory tiling for attention.
**Acceptable tradeoffs**: None this iteration — pure optimization pass.
```

This is the architect's memory across iterations. It reads the full log before planning each iteration and writes its entry before creating tickets.

### The loop cycle

1. Architect reads: goal + iteration log + feedback output from previous iteration
2. Architect writes iteration log entry (assessment of previous results, strategy for this iteration, acceptable tradeoffs)
3. Architect creates/updates areas and tickets for this iteration
4. Managers + coders work the tickets
5. Review/merge as normal
6. Queue drains
7. Orchestrator runs the feedback command
8. Back to step 1

### Soft gating via architect-declared tradeoffs

When the architect plans an iteration, it declares **expected tradeoffs** in the iteration log. For example: "pathfinding refactor may drop score up to 2%, this is acceptable because eval speed matters more right now."

After the feedback step, the architect reviews actual results against its declared expectations. If reality is within the declared tolerance, it continues. If reality diverges significantly (expected 2% drop, got 15%), the architect's next iteration should be "investigate what went wrong" rather than pressing forward.

This keeps gating as **architect judgment** rather than hard numeric thresholds. The architect knows the strategy and can weigh multi-dimensional tradeoffs (score vs eval time, correctness vs performance, etc).

For cases with hard constraints (GPU kernel: "output must not change"), the architect encodes that in its strategy: "correctness is non-negotiable, any output difference means revert."

### How it covers each example case

**GPU kernel improver**: Feedback is `make bench`, goal says "maximize t/s, output must be identical." Architect iterates on kernel optimizations, tracks t/s progression in the log. Straightforward metric chase with a hard correctness constraint.

**Minecraft bot**: Feedback could be `make test-survival` or tailing a game log. Goal is qualitative: "survive the first night, then build a house." Architect evaluates from logs/observations, plans next improvements. No single score — pure judgment-driven iteration.

**Metta CoGames bot**: Feedback is `make upload && sleep 300 && make fetch-scores`. Goal describes the competitive context. Architect can choose foundation phases (faster pathfinding, accepting small score dips) vs optimization phases (chase the leaderboard score), declaring tradeoffs explicitly each iteration.

### Validation during review

The review agent runs `make test` as it already does — per-PR validation is normal CI. The loop feedback command runs after the full iteration merges. Per-PR checks catch breakage; the feedback step is a coarser "how did the whole iteration go" assessment that feeds back into the architect's planning.

### What this MVP deliberately does not include

These are all potential future iterations, not MVP scope:

- A/B branching or parallel strategy testing
- Automatic rollback of bad iterations
- Structured metric parsing or numeric threshold config
- Special review agent behavior for loop mode
- Manager-level gating or per-ticket scoring
- Strategy phase declarations as a formal config concept

The MVP is three moving parts: **feedback command + iteration log + architect judgment**.
