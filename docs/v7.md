# v7 plan

Focus: code quality, correctness, and test coverage.

## Split orchestrator.nim into modules

orchestrator.nim is 3,878 lines with ~197 procs covering everything from git
operations to prompt building to the main event loop. This makes it difficult
to test individual subsystems in isolation and increases the risk of unintended
coupling.

Natural module boundaries identified:

| Proposed module | Responsibility | Approx procs |
|-----------------|---------------|--------------|
| `git_ops.nim` | gitRun, gitCheck, worktree add/remove/recover | ~10 |
| `lock_management.nim` | repo lock, stale PID detection, plan worktree lock | ~10 |
| `ticket_metadata.nim` | parse area/depends/worktree fields, ticket/area ID helpers, open/done listing | ~15 |
| `prompt_builders.nim` | all buildXxxPrompt procs, plan scope, prediction prompt | ~15 |
| `output_formatting.nim` | formatAgentRunNote, formatMetricsNote, formatDuration, truncation | ~20 |
| `ticket_analysis.nim` | prediction, post-analysis, difficulty classification | ~12 |
| `merge_queue.nim` | ensureMergeQueueInitialized, enqueueMergeRequest, processMergeQueue | ~8 |
| `coding_agent.nim` | executeAssignedTicket, agent thread pool, slot management | ~10 |
| `architect_agent.nim` | spec sync, area generation, architect should-run check | ~12 |
| `manager_agent.nim` | ticket generation from areas, batch prompt | ~8 |
| `ticket_assignment.nim` | assignOldestOpenTicket, dependency check, worktree cleanup | ~12 |
| `health_checks.nim` | master health, health cache, state invariant validation | ~8 |
| `mcp_server.nim` | createOrchestratorServer, submit_pr/submit_review handlers, shared buffers | ~5 |
| `shared_state.nim` | locks init, token budget, rate limiting, session stats | ~15 |
| `interactive_sessions.nim` | interactive plan/ask sessions, input helpers | ~6 |
| `orchestrator.nim` (remaining) | main loop, signal handlers, runOrchestrator entry points | ~10 |

Approach:
- Extract one module at a time, bottom-up (start with modules that have no
  internal dependencies: git_ops, lock_management, ticket_metadata)
- Each extraction should be a single commit that moves procs and updates imports
- Existing tests must pass after each extraction (no behavior changes)
- Add focused unit tests for newly isolated modules where coverage is thin

## Fix SIGSEGV on shutdown

After the orchestrator finishes its main loop, it calls `httpServer.close()`
from the main thread while mummy's `serve()` is still running on the server
thread. The `close()` triggers `loopForever` to exit which calls `destroy`,
and the crash happens in `addToSharedFreeList` inside Nim's allocator — a
use-after-free or thread-safety issue in mummy's shutdown path.

Traceback from docs/integration_issues.md:
```
orchestrator.nim runHttpServer
mcp_server_http.nim serve
mummy.nim serve → loopForever → destroy
alloc.nim dealloc → rawDealloc → addToSharedFreeList
SIGSEGV: Illegal storage access. (Attempt to read from nil?)
```

Options to investigate:
1. Coordinate shutdown by setting a flag and letting the server thread close
   itself (avoid cross-thread close)
2. Check if newer mummy versions fix the issue (currently pinned to 0.4.7)
3. Add a small sleep between `close()` and `joinThread()` to let the server
   thread drain (band-aid, not a real fix)
4. Switch to a different HTTP server library if mummy's threading model is
   fundamentally incompatible with ARC

The crash currently happens after tests pass so it doesn't block functionality,
but it means the orchestrator never exits cleanly.

## Ticket dependency cycle detection

Ticket dependencies (`**Depends:** 0042, 0043`) are checked at assignment time:
`dependenciesSatisfied` returns false if any dependency isn't in the done set.
But there is no cycle detection. If ticket A depends on B and B depends on A,
both remain in `open/` forever with no error or warning logged.

Changes needed:
- At ticket creation time (manager), validate that new dependency references
  don't form cycles. Build a directed graph from all open+in-progress tickets
  and check for strongly connected components.
- At orchestrator startup or periodically during the main loop, scan for
  tickets that can never be assigned due to circular dependencies and log a
  warning (or move them to stuck/).
- Consider adding a `scriptorium status` output that flags blocked tickets
  and explains why (missing deps, circular deps).

## Review agent test coverage

The review agent has good integration-level test coverage (approve flow,
request_changes flow, stall-defaults-to-approve) but these all use a fake
`noopRunner`. What's not tested:

- `buildReviewAgentPrompt` output structure — no test verifies the rendered
  prompt contains the expected sections (diff, area context, submit summary)
- Diff generation edge cases — what happens when the diff is empty, or when
  the branch has been force-pushed and `git diff master...branch` fails?
- Review feedback truncation boundary — `ReviewFeedbackMaxBytes` (4096) is
  defined but there's no test that exercises the truncation path
- Real model execution — integration tests with a live model to verify the
  review agent actually calls `submit_review` (similar to existing
  `integration_mcp_submit_pr_live.nim` but for the reviewer)
