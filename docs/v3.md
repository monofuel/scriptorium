## Design

All v3 metrics should be visible in `scriptorium run` logs (INFO level to stdout) so you can watch the system in real time. The goal is that tailing the log gives you a clear picture of what agents are doing, how long things take, and whether work is succeeding or failing.

### Tick summary line

At the end of each orchestrator tick, log a single summary line at INFO level:

```
[INFO] tick 42 summary: architect=no-op manager=no-op coding=0031(running, 3m12s) merge=idle open=2 in-progress=1 done=14
```

This gives a snapshot of the entire system state in one line. Fields:
- `architect` / `manager`: `no-op`, `updated`, or `skipped`
- `coding`: ticket ID + status (`running`, `stalled`, `submitted`, `failed`) + wall time
- `merge`: `idle`, `processing`, or ticket ID being merged
- `open` / `in-progress` / `done`: ticket counts

### Ticket lifecycle logging

Log at INFO level for every state transition, with timing:

```
[INFO] ticket 0031: open -> in-progress (assigned, worktree=/tmp/scriptorium/.../0031)
[INFO] ticket 0031: coding agent started (model=claude-opus-4-6, attempt 1/2)
[INFO] ticket 0031: coding agent finished (exit=0, wall=4m32s, stall=false)
[INFO] ticket 0031: submit_pr called (summary="add retry logic")
[INFO] ticket 0031: merge queue entered (position=1)
[INFO] ticket 0031: merge started (make test running)
[INFO] ticket 0031: merge succeeded (test wall=45s)
[INFO] ticket 0031: in-progress -> done (total wall=5m17s, attempts=1)
```

For failures:
```
[INFO] ticket 0032: coding agent stalled (attempt 1/2, no submit_pr)
[INFO] ticket 0032: make test before retry: FAIL (exit=1, wall=30s)
[INFO] ticket 0032: continuation prompt sent (attempt 2/2, test_status=failing)
[INFO] ticket 0032: in-progress -> open (reopened, reason=stall-exhausted, attempts=2, total wall=8m41s)
```

### Metrics to track per ticket

Capture these in the agent run notes (already written to the plan branch) so they persist:

- **wall_time_seconds**: total time from assignment to done/reopen
- **coding_wall_seconds**: time spent in coding agent (per attempt and total)
- **test_wall_seconds**: time spent running make test (merge queue + stall checks)
- **attempt_count**: number of coding agent attempts
- **outcome**: `done`, `reopened`, `parked`
- **failure_reason**: `stall`, `test_failure`, `merge_conflict`, `timeout_hard`, `timeout_no_output`, `parked` (if not done)
- **model**: which model was used
- **stdout_bytes**: size of agent stdout (proxy for token usage until we can get real counts)

### Session summary on shutdown

When the orchestrator shuts down (signal or idle exit), log a session summary:

```
[INFO] session summary: uptime=1h23m ticks=47 tickets_completed=3 tickets_reopened=1 tickets_parked=0 merge_queue_processed=3
[INFO] session summary: avg_ticket_wall=5m12s avg_coding_wall=4m02s avg_test_wall=38s first_attempt_success=75%
```

### `scriptorium status` enhancement

`scriptorium status` currently shows ticket counts. Extend it to show:
- per-ticket timing (how long has the in-progress ticket been running?)
- last N completed tickets with outcome and wall time
- first-attempt success rate across all done tickets
