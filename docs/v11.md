# v11 plan

Focus: make scriptorium work well on fresh Nim projects out of the box.

## Background

Scriptorium has been developed and tested almost exclusively on itself. When
pointed at a fresh repo — even a Nim project — the user has to manually create
an AGENTS.md, a Makefile with the right targets, and a scriptorium.json before
the orchestrator can function. There is no guidance, no validation, and no
scaffolding. This version closes those gaps so that `scriptorium init` on a new
Nim repo produces a working setup with minimal manual intervention.

We are staying Nim-only and keeping the existing `make test` /
`make integration-test` / `make e2e-test` quality gate convention.

## Project opinions

Scriptorium is opinionated about how managed projects are structured. These
opinions keep the orchestrator simple and predictable. They may evolve over
time, but for now they are rigid. Someone who wants to use scriptorium to
manage a video game, an analytics service, or any other Nim project should
expect these constraints.

### Language & dependency management

- Nim is the implementation language.
- nimby is the dependency manager (`nimby.lock` for pinned deps, `nimby sync`).
- A `.nimble` file provides package metadata and versioning.

### Project structure

- `src/` for source code.
- `tests/` for all test files (unit, integration, e2e).
- `Makefile` at repo root with required targets.
- `AGENTS.md` at repo root for coding agent instructions.
- `scriptorium.json` at repo root for orchestrator configuration.
- `docs/` for project documentation and version plans.

### Make targets (quality gates)

The orchestrator uses Make targets as quality gates. These are the contract
between the project and scriptorium:

- `make build` — compile the project and produce artifacts. Must exit 0 on
  success. Run before tests.
- `make test` — unit tests. Must be fast, require no network access, and no
  API keys. Run on every commit and before every merge to the default branch.
  Failure is a hard gate — code does not merge.
- `make integration-test` — tests that hit real services (APIs, databases,
  external tooling). May cost money and require credentials. Run on default
  branch pushes. Failure reopens the ticket.
- `make e2e-test` — (optional) full end-to-end smoke tests. Heaviest and
  most expensive. Run on default branch pushes if present. Failure reopens the
  ticket.

A fresh project generated by `scriptorium init` includes placeholder targets
that exit 0. As the coding agent adds code, tickets should instruct it to
replace placeholders with real build and test commands.

### Default branch detection

Scriptorium does not hardcode `master` or `main`. Instead it resolves the
default branch dynamically:

1. Check `refs/remotes/origin/HEAD` — if set, use that.
2. If not set, probe for common names (`master`, `main`, `develop`) and use
   whichever exists locally.
3. During `scriptorium init`, once resolved, run
   `git remote set-head origin <branch>` so future runs skip the probe.
4. If nothing works, error with a clear message asking the user to set the
   default branch.

This supports any branch naming convention without additional configuration.

## 1. `scriptorium init` generates AGENTS.md

Currently `scriptorium init` creates the plan branch but leaves master
untouched. The coding agents all read `AGENTS.md` for project guidance, but
none is generated.

- On init, copy `agents_example.md` to `AGENTS.md` on master and commit it.
- The example is already a good generic Nim template. It covers style, imports,
  error handling, and testing conventions.
- If `AGENTS.md` already exists, skip this step (don't overwrite user work).
- Print a note: `Created AGENTS.md — edit to match your project conventions.`

Files: `src/scriptorium/init.nim`, `src/scriptorium/prompts/agents_example.md`

## 2. `scriptorium init` generates a starter Makefile

The orchestrator and merge queue hardcode `make test` and
`make integration-test` as quality gates. A fresh repo has no Makefile, so the
orchestrator immediately fails.

- On init, if no Makefile exists on master, generate a minimal one:
  ```makefile
  test:
  	@echo "no tests yet"

  integration-test:
  	@echo "no integration tests yet"

  e2e-test:
  	@echo "no e2e tests yet"
  ```
- Commit it on master with a clear message.
- If a Makefile already exists, skip (don't overwrite).
- Print a note: `Created Makefile with placeholder targets — replace with real
  build commands.`

The placeholder targets exit 0 so the orchestrator can run immediately. As the
coding agent adds code, the spec or tickets should instruct it to add real test
commands.

Files: `src/scriptorium/init.nim`

## 3. `scriptorium init` generates a starter scriptorium.json

The orchestrator reads `scriptorium.json` for agent model/harness config and
the MCP endpoint. Without it, everything falls back to defaults which may not
match the user's setup (e.g. they may not have an OpenAI key).

- On init, if no `scriptorium.json` exists, generate one with `defaultConfig()`
  and write it to the repo root.
- Do not commit it — it may contain user-specific preferences and should be
  gitignored or committed at the user's discretion.
- Print a note: `Created scriptorium.json — configure agent models and
  harnesses.`

Files: `src/scriptorium/init.nim`, `src/scriptorium/config.nim`

## 4. Pre-flight validation in `scriptorium run`

The orchestrator currently starts and then fails cryptically if prerequisites
are missing. Add upfront checks before launching:

- Verify `scriptorium/plan` branch exists (already done).
- Verify `AGENTS.md` exists on master. Warn if missing.
- Verify Makefile exists on master. Error if missing (orchestrator cannot
  function without quality gates).
- Verify `make test` and `make integration-test` targets exist (run
  `make -n test` and `make -n integration-test` dry-run). Warn if either fails.
- Verify agent binary is available (`codex`, `claude`, or `typoi` depending on
  configured harness). Error if missing.
- Verify agent auth is available (API keys or OAuth credentials for the
  configured harness). Error if missing.

Print clear, actionable error messages:
```
Error: Makefile not found. Run `scriptorium init` or create a Makefile with
`test` and `integration-test` targets.
```

Files: `src/scriptorium/orchestrator.nim` (or new `src/scriptorium/preflight.nim`)

## 5. Improve post-init guidance

The current post-init output is:
```
Next steps:
  scriptorium plan   — build your spec with the Architect
  scriptorium run    — start the orchestrator
```

Expand this to guide the user through the full setup:
```
Initialized scriptorium workspace.

Created:
  AGENTS.md           — project conventions for coding agents (edit to fit)
  Makefile            — placeholder test targets (replace with real commands)
  scriptorium.json    — agent configuration (set your models and API keys)
  scriptorium/plan    — plan branch with spec, areas, and tickets

Next steps:
  1. Edit scriptorium.json to configure your agent models and harnesses
  2. Edit AGENTS.md to describe your project conventions
  3. Run `scriptorium plan` to build your spec with the Architect
  4. Run `scriptorium run` to start the orchestrator
```

Files: `src/scriptorium/init.nim`

## 6. Spec prompt should reference AGENTS.md

The architect prompt already reads AGENTS.md, but the spec placeholder created
by init is generic. When the user runs `scriptorium plan` on a fresh repo, the
architect should be aware it's starting from scratch.

- Update the spec placeholder to note that this is a new project:
  ```
  # Spec

  This is a new project. See AGENTS.md for project conventions.

  Run `scriptorium plan` to build your spec with the Architect.
  ```

Files: `src/scriptorium/init.nim`

## 7. Update e2e test to use `scriptorium init`

The existing Euler e2e test pre-configures the repo with a Makefile and
AGENTS.md before running. Update it to stop seeding those files and rely on
`scriptorium init` instead:

- Remove manual AGENTS.md and Makefile creation from test setup.
- Run `scriptorium init` (programmatically via `runInit`) on the temp repo.
- Verify AGENTS.md, Makefile, and scriptorium.json were created by init.
- Continue with the existing flow: write spec, start orchestrator, verify
  ticket completion and `make test` passing.

This proves init produces a repo the orchestrator can run against without
any manual scaffolding.

Files: `tests/e2e_euler.nim` (or equivalent), `tests/support/live_integration_support.nim`

## 8. Default branch detection

Replace all hardcoded `master` references in the orchestrator, merge queue,
and init code with a resolved default branch.

- Implement `resolveDefaultBranch()` using the detection logic described in
  the project opinions section above.
- `scriptorium init` calls `git remote set-head origin <branch>` after
  resolving, so subsequent runs are fast.
- Update orchestrator, merge queue, and worktree management to use the
  resolved branch instead of a hardcoded string.
- Add a unit test for the resolution logic (mock git output).

Files: `src/scriptorium/git_utils.nim` (or similar), `src/scriptorium/init.nim`,
`src/scriptorium/orchestrator.nim`

## 9. Change `scriptorium --init` to `scriptorium init`

The init command currently uses `--init` flag syntax while every other command
uses subcommand syntax (`run`, `plan`, `ask`, `status`, `worktrees`). Change
it to `scriptorium init` for consistency.

- Accept `scriptorium init [path]` as the new syntax.
- Keep `--init` as a deprecated alias for backwards compatibility (or just
  drop it — there are no external users yet).
- Update CLI help, README, and all documentation references.

Files: `src/scriptorium.nim` (CLI parsing), `README.md`
