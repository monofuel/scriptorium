# v15 plan

Focus: replace the locking model with transactional commit locks and an
orchestrator singleton guard.

## Background

The current locking model uses a `mkdir`-based lock directory
(`.scriptorium/locks/repo.lock`) that is held for the duration of plan
branch operations. Problems discovered in v14:

- **Stale locks after crash**: When the orchestrator is killed (SIGKILL,
  OOM, `docker compose down` timeout), the lock directory is not cleaned up.
  The next startup crashes with `another planner/manager is active`.
- **Long-held locks**: Some code paths hold the lock during agent execution
  or multi-step operations, which is unnecessary since post-v13 all plan
  branch writes happen on the main thread.
- **No staleness detection**: The lock has no PID, timestamp, or TTL — if
  it exists, it's treated as live regardless of age.
- **Recovery requires manual cleanup**: The operator must delete
  `.scriptorium/locks/repo.lock` and prune worktrees before restarting.

## Design

Two separate mechanisms replace the current single lock:

### 1. Orchestrator singleton guard

Prevents two `scriptorium run` instances from operating on the same repo.

- On startup, write `.scriptorium/orchestrator.pid` containing the process
  PID and a startup timestamp.
- If the file already exists, read the PID and check liveness with
  `kill(pid, 0)`.
  - If the PID is alive: abort with a clear error message.
  - If the PID is dead: log a warning, overwrite the PID file, proceed.
- On clean shutdown, delete the PID file.
- This is **not** a lock — it's a one-time startup check. It is not
  acquired/released during operation.
- The PID check does not work across container boundaries (different PID
  namespaces). This is acceptable — the constraint is that scriptorium runs
  either always in a container or always on the host for a given repo, never
  both simultaneously.

### 2. Transactional commit lock

Short-lived lock held only for the duration of a read-modify-commit cycle
on the plan branch.

**Contract:**
- The lock is acquired immediately before a plan branch write operation.
- The lock holder reads state, modifies files, runs `git commit`, then
  releases the lock.
- The lock is **never** held during agent execution, API calls, test runs,
  or anything that takes more than a few seconds.
- Maximum expected hold time: < 2 seconds (a `git add` + `git commit`).

**Implementation:**
- Lock file: `.scriptorium/commit.lock` (a regular file, not a directory).
- On acquisition, write a JSON object: `{"pid": <pid>, "timestamp": <unix_epoch>}`.
- On release, delete the file.
- Staleness threshold: 30 seconds. If the lock file exists and the timestamp
  is older than 30 seconds, the lock is considered stale.

**Contention handling:**
- If the lock file exists and is fresh (< 30s old): wait briefly (100ms)
  and retry, up to N retries (e.g. 50 retries = 5 seconds max wait).
- If the lock file exists and is stale (>= 30s old): log a warning
  ("stealing stale commit lock from PID <pid>, held for <duration>"),
  delete the lock file, and acquire.
- If all retries are exhausted: fail with a clear error. This should never
  happen in practice — a 5-second wait means something is seriously wrong.

**Callers:**
- Orchestrator: architect area writes, manager ticket writes, area hash
  updates, ticket state transitions, merge queue operations, health cache
  writes.
- `scriptorium plan`: spec.md commits at the end of each interactive turn
  or one-shot invocation.
- These are all brief git commit operations. No caller holds the lock for
  more than the time it takes to stage and commit files.

### What changes

| Current | New |
|---------|-----|
| `withRepoLock()` — mkdir-based, no timeout, no staleness | `withCommitLock()` — file-based, 30s TTL, staleness detection |
| `withLockedPlanWorktree()` — acquires repo lock + ensures worktree | Split: `ensurePlanWorktree()` (no lock) + `withCommitLock()` (brief) |
| No orchestrator singleton check | `.scriptorium/orchestrator.pid` on startup |
| Lock held for entire `withLockedPlanWorktree` callback | Lock held only for git add + commit |
| Stale lock requires manual `rm -rf` | Stale lock auto-detected and stolen after 30s |

## Implementation plan

### 1. Implement transactional commit lock

Create a new `commit_lock.nim` module (or refactor `lock_management.nim`):

- `acquireCommitLock(repoPath): bool` — write lock file with PID +
  timestamp. Handle contention with retry + staleness detection.
- `releaseCommitLock(repoPath)` — delete lock file.
- `withCommitLock(repoPath, callback)` — acquire, run callback, release.
  Guarantee release even on exception (try/finally).
- The lock file path: `repoPath / ".scriptorium" / "commit.lock"`.

### 2. Implement orchestrator singleton

Add to orchestrator startup (`runOrchestrator` or `runOrchestratorLoop`):

- `acquireOrchestratorPid(repoPath)` — write PID file, check for existing
  live instance.
- `releaseOrchestratorPid(repoPath)` — delete PID file on shutdown.
- Wire into signal handler cleanup.

### 3. Refactor callers

Split `withLockedPlanWorktree` into two concerns:

- `ensurePlanWorktree(repoPath)` — ensure the plan worktree exists and is
  checked out. No locking needed (idempotent, safe to call concurrently
  since git worktree operations are atomic).
- `withCommitLock(repoPath, callback)` — brief lock for the commit
  operation only.

Update all call sites in:
- `orchestrator.nim` — manager completion handler, ticket transitions,
  health cache writes.
- `architect_agent.nim` — area file writes, spec hash updates.
- `merge_queue.nim` — ticket state transitions, queue metadata writes.
- `interactive_sessions.nim` — plan session commits.
- `recovery.nim` — orphaned worktree cleanup (may not need locking at all
  if it only runs at startup before the orchestrator loop).

### 4. Remove old locking

- Delete `withRepoLock()` and `withLockedPlanWorktree()`.
- Remove `.scriptorium/locks/` directory handling.
- Remove any lock cleanup from `recoverFromCrash()`.

### 5. Update tests

- Unit tests for commit lock: acquisition, release, contention, staleness
  detection, retry behavior.
- Unit tests for orchestrator PID: normal startup, stale PID detection,
  live PID rejection.
- Update existing tests that use `withLockedPlanWorktree` to use the new
  API.
- Integration test: start orchestrator, try to start a second one, verify
  it fails with a clear error.

### 6. Update documentation

- Update `AGENTS.md` if it references the locking model.
- Update `spec.md` section 17 (Plan Branch Locking) to describe the new
  transactional model.

## Platform compatibility

| Platform | Singleton PID | Commit lock | Notes |
|----------|--------------|-------------|-------|
| Linux local | `kill(pid,0)` | file write + timestamp | Full support |
| macOS local | `kill(pid,0)` | file write + timestamp | Full support |
| Linux NFS | `kill(pid,0)` | file write + timestamp | Timestamp-based, no kernel lock dependency |
| Cross-container | No (PID namespaces differ) | Yes (timestamp-based) | Acceptable — single execution context per repo |

## Risks

- **Refactoring surface area**: Many call sites use `withLockedPlanWorktree`.
  Each needs to be audited and split into worktree-ensure + commit-lock.
  This is mechanical but easy to get wrong.
- **Lock ordering**: If any code path acquires the commit lock and then does
  something slow (agent call, test run), we're back to the original problem.
  Need to audit all `withCommitLock` callbacks to ensure they're brief.
- **Clock skew on NFS**: The staleness threshold depends on filesystem
  timestamps. NFS clock skew could cause premature or delayed staleness
  detection. Mitigation: use a generous threshold (30s) that tolerates
  typical NFS skew (< 1s).

## Resolved without implementation

The transactional commit lock (`withCommitLock`) was the original core design
goal but is no longer needed. Each problem it was designed to solve was fixed
by other means:

- **Stale locks after crash** → fixed by PID liveness checks in `withRepoLock`
  plus cross-container detection (same-PID treated as stale).
- **Long-held locks during agent execution** → already fixed in v13 when all
  plan branch writes moved to the main thread.
- **No staleness detection** → fixed by PID-based staleness in `lockPathIsStale`.
- **Plan branch corruption** → fixed by persistent plan worktree in v15.1,
  eliminating the ephemeral create/destroy churn that corrupted git metadata.

The current `withRepoLock` + `withLockedPlanWorktree` pattern is heavier than
a minimal commit lock but is correct, tested, and stable. Refactoring 20+ call
sites for a marginal improvement is not worth the risk.

## Post-analysis

### What went well

- **Parallel integration tests** cut merge queue wall time from ~18 min to
  ~6 min per merge. This change alone made the v15 run viable.
- **Review agent caught real issues**: ticket 0074 was rejected on first
  attempt for including unrelated test changes. The second attempt was clean.
- **First-attempt success rate**: 73% (22/30), consistent with v14.

### Problems encountered

- **Unit tests spawned real `claude` processes**: Two tests in
  `test_orchestrator_flow.nim` called `runOrchestratorForTicks` without a
  fake runner, causing `make test` to hang indefinitely in the container.
  One test used a fake-codex-on-PATH approach that didn't work because the
  harness picks `claude-code`, not `codex`. Fixed by replacing both with
  proper `fakeRunner` procs.

- **Stale lock after crash**: Killing the orchestrator mid-run left
  `.scriptorium/locks/repo.lock` behind, blocking restart. This is the
  exact problem v15's commit lock was designed to solve — ironic that it
  blocked the run that was supposed to fix it. Required manual `rm -rf`.

- **Health cache invalidation**: Every commit to master invalidates the
  health cache, triggering a full `make test` + `make integration-test`
  (~20 min). Seeding the cache manually after each fix was tedious.

- **Plan branch transition failures**: After successful merges, the
  orchestrator's completion handler failed with "cannot open" errors when
  trying to move tickets from `in-progress/` to `done/` on the plan
  branch. This left tickets stuck as in-progress despite being merged to
  master. Required manual reconciliation on the plan branch between runs.
  Root cause: likely a race between the merge completing and the plan
  worktree state being stale.

- **Merge conflicts on concurrent PID guard tickets**: Ticket 0085
  (duplicate PID guard implementation) hit merge conflicts twice because
  0084 had already merged the same feature. The orchestrator correctly
  reopened and retried, but it wasted time.

### Scope reduction

The transactional commit lock (`withCommitLock`) was the core design goal
but was not implemented. The orchestrator created tickets for the PID guard
and related work but did not generate tickets for the commit lock refactor,
likely because it's a large multi-file refactoring that touches 20+ call
sites. The PID guard alone provides meaningful improvement — stale PID
files are auto-detected on startup, unlike the old mkdir-based lock.
