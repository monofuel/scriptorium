# Ticket Difficulty Prediction And Post-Analysis

V3 feature: predict ticket difficulty before coding and compare predictions to actuals after completion.

## Scope

- Ticket difficulty prediction (§18):
  - Before a ticket is assigned to a coding agent, run a prediction prompt.
  - Prediction generated by a lightweight agent call using the configured coding agent model via codex harness.
  - Prediction prompt includes: ticket content, relevant area content, current spec summary context.
  - Prediction output fields:
    - `predicted_difficulty`: `trivial`, `easy`, `medium`, `hard`, or `complex`.
    - `predicted_duration_minutes`: estimated wall time in minutes.
    - `reasoning`: brief explanation of the prediction.
  - Prediction logged at INFO level: `ticket <id>: predicted difficulty=<level> duration=<n>min`.
  - Prediction appended to ticket markdown as a prediction section before coding begins.
  - Prediction must not block or significantly delay ticket assignment (should be fast).

- Ticket post-analysis (§19):
  - After a ticket reaches `done` or is reopened/parked, run a brief post-analysis.
  - Compares predicted vs actual metrics:
    - Predicted difficulty vs actual attempt count and outcome.
    - Predicted duration vs actual wall time.
  - Post-analysis output fields:
    - `actual_difficulty`: assessment based on actual metrics (`trivial`, `easy`, `medium`, `hard`, `complex`).
    - `prediction_accuracy`: `accurate`, `underestimated`, or `overestimated`.
    - `brief_summary`: one-sentence summary of what happened.
  - Logged at INFO level: `ticket <id>: post-analysis: predicted=<level> actual=<level> accuracy=<accuracy> wall=<duration>`.
  - Appended to ticket markdown as a post-analysis section.

## V3 Known Limitations

- Predictions use heuristic difficulty levels, not calibrated estimates — accuracy improves over time.
- Post-analysis is a simple predicted-vs-actual comparison, not detailed root-cause analysis.
- Dynamic routing based on predicted difficulty is out of scope for V3.
